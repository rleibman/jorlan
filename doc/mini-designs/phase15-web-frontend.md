# Phase 15 Mini-Design: Web Frontend

Date: 2026-06-10  
Status: Draft

---

## Goals

Deliver a browser-based UI equivalent of the shell:

- A chat interface (prompt input + streaming response display)
- Full configuration screens covering all shell capabilities
- Agent session management, approvals, memory, scheduler, event log, users/roles, and skill registry

---

## Technology Stack

| Concern | Choice | Rationale |
|---------|--------|-----------|
| Language | Scala.js (same as shell) | Shares model types directly |
| UI framework | React 19 via `japgolly/scalajs-react` 4.0.0 | Same as dmscreen; battle-tested |
| Component library | MUI (Material UI) v9 via ScalablyTyped | Replaces SemanticUI from dmscreen; already wired in `stLib/build.sbt` |
| GraphQL client | Caliban `caliban-client` (JS) | Already in `commonWeb` settings; same caliban schema as shell |
| HTTP/WS transport | sttp + ZIO | Same pattern as shell's `GraphQLClient` |
| Bundler | Webpack 5 via sbt-scalajs-bundler | Same as dmscreen |
| ScalablyTyped wrappers | `stLib/` sub-project | Separate project to avoid polluting main build; same pattern as dmscreen |
| Auth | `zio-auth` `AuthClient.whoami` + JWT cookie | Same mechanism as dmscreen |
| Routing | Hash-based router in `AppRouter.scala` | Single-page app; no server-side routing needed |

## Module Structure

### `stLib/` (separate SBT project, already created)

Generates ScalablyTyped Scala wrappers for:
- `react` / `react-dom` (v19)
- `@mui/material` (v9)
- `@emotion/react` / `@emotion/styled` (v11)
- `csstype`

Output package: `net.leibman.jorlan`  
Published locally as `net.leibman:jorlan-stlib:1.0.0`.

### `web/` (SBT module in main build, already created)

Depends on: `model`, `stlib` (published), `caliban-client`, `zio`, `sttp`.

Source root: `web/src/main/scala/jorlan/web/`  
Static assets: `web/src/main/web/` (HTML, CSS, images, favicon)

Build targets:
- `web/debugDist` — fast-opt JS → `debugDist/`
- `web/dist` — full-opt JS → `dist/`

The `dist/` directory is already mapped into the Debian package by `debianSettings`.

## Application Architecture

### Entry Point

```
JorlanWebApp.scala
  └── ScalaFnComponent
        ├── useEffect: AuthClient.whoami → Option[User]
        └── render:
              ├── None → LoginRouter (zio-auth)
              └── Some(user) → AppRouter(user)
```

### Routing (hash-based)

| Route | Component | Description |
|-------|-----------|-------------|
| `#/` | `ChatPage` | Main chat interface |
| `#/sessions` | `SessionsPage` | Session list + create/stop |
| `#/approvals` | `ApprovalsPage` | Approval queue |
| `#/memory` | `MemoryPage` | Memory browser |
| `#/scheduler` | `SchedulerPage` | Job management |
| `#/events` | `EventLogPage` | Event log browser |
| `#/skills` | `SkillsPage` | Skill registry |
| `#/users` | `UsersPage` (admin) | User/role management |
| `#/settings` | `SettingsPage` | Personality + model config |

All routes render inside a shared `AppShell` (MUI `AppBar` + `Drawer` + `main` content area).

### GraphQL Client Wiring

```
JorlanClient.scala  (generated by Caliban CLI, like dmscreen's DND5eClient.scala)
  │
  └─ ScalaJSClientAdapter (borrowed from dmscreen pattern)
        ├── query/mutation: sttp POST to /api/graphql
        └── subscription: WebSocket to ws[s]://host/api/subscriptions
```

The Caliban client schema file is generated via `sbt server/calibanGenClient` and checked into
`web/src/main/scala/jorlan/graphql/client/JorlanClient.scala`.

### State Management

Each page component manages its own state via `useState`/`useEffect` hooks. No global state store.
Shared data (current user, session ID) is passed as props from `AppRouter`.

Approval notifications and event log tail use GraphQL subscriptions (WebSocket), wired in the
relevant page component's `useEffect`.

## Page Designs

### ChatPage

```
┌─────────────────────────────────────────────────────┐
│  AppBar: Jorlan  [Session: #42 | gpt-4o]  [New]     │
├─────────────────────────────────────────────────────┤
│                                                     │
│  [10:32:01]  ⚙  Session started                    │
│  [10:32:15]  ❯  What's the weather in London?      │
│  [10:32:16]  ✦  The weather in London is ...       │  ← scrollable MUI Box
│                                                     │
├─────────────────────────────────────────────────────┤
│  ❯ [TextField multiline, full width]    [Send]      │
└─────────────────────────────────────────────────────┘
```

- Uses `submitMessage` mutation; response streamed via `messageStream` subscription
- Shift+Enter = newline, Enter = send
- New session button calls `createSession` mutation
- Model selector in AppBar secondary action

### ApprovalsPage

Live list via `approvalNotifications` subscription. Each row: capability requested, agent,
scope, Approve/Deny buttons. `decideApproval` mutation on click.

### MemoryPage

Table with search box. Columns: scope badge, content, created-at. Actions: Forget, Mark
Shared/Private. `listMemory` query, `forgetMemory` / `markMemoryShared` / `markMemoryPrivate`
mutations.

### SchedulerPage

Jobs table: name, status, next-run, actions (pause/resume/cancel/run-now/delete).
Triggers sub-table. Create-job dialog with trigger form.

### EventLogPage

Filterable table (session ID, actor, event type). Paginated. Live tail toggle using
`eventLogTail` subscription.

### UsersPage

Users table with edit/deactivate. Roles assignment dialog. Permission grant/revoke. Admin-only
(gated by capability check client-side + server enforces it).

### SettingsPage

Personality editor (formality select + freeform fields). Model selector for default model.

## Build & Packaging

### Development Workflow

```bash
# Terminal 1 – run the server
cd /home/rleibman/projects/jorlan
sbtn server/run

# Terminal 2 – compile and bundle web in watch mode
cd /home/rleibman/projects/jorlan
sbtn ~web/fastOptJS/webpack
cp -r web/src/main/web/* debugDist/
```

Or use the existing `web/debugDist` task:
```bash
sbtn web/debugDist
```

### Production Build

```bash
sbtn web/dist        # full-opt JS → dist/
sbtn server/debian:packageBin  # includes dist/ contents via debianSettings
```

### stLib Publish Workflow

The `stLib/` sub-project must be published to local Maven before the main build can compile:

```bash
cd stLib
sbtn publishLocal   # publishes net.leibman:jorlan-stlib:1.0.0
```

This is already the pattern in dmscreen. The CI pipeline must run `stLib/publishLocal` before
the main build step.

### CI Integration

Add a build stage to GitHub Actions (when created in Phase 18):
```yaml
- name: Publish stLib
  run: cd stLib && sbt publishLocal
- name: Build web
  run: sbt --error web/dist
- name: Package server (includes web dist)
  run: sbt --error server/debian:packageBin
```

## Static File Serving (Server Changes)

Jorlan serves the web frontend directly — no reverse proxy or separate nginx needed.

### Server-side: StaticFileRoutes

Add `server/src/main/scala/jorlan/web/StaticFileRoutes.scala`:

```scala
Routes(
  // SPA: serve index.html for any unknown path so deep-linking works
  Method.GET / trailing -> Handler.fromFileTree(webRootPath)
    .orElse(Handler.fromFile(webRootPath / "index.html"))
)
```

These routes are added to `Jorlan.zapp` **before** the auth/GraphQL routes (lowest priority,
but as catch-all for `GET`).

### Configuration

Add to `application.conf`:
```hocon
jorlan.web.root = "/opt/jorlan/www"
```

For local development, override via `.env` or local `application.conf`:
```hocon
jorlan.web.root = "debugDist"
```

If the configured path does not exist, the static file routes return `404` without crashing.

### Configuration Model

Add `webRoot: String` to `JorlanAppConfig` (in `configuration.scala`) and read via `zio-config`.

## Installer Notes

### Debian Package Changes

The `debianSettings` in `build.sbt` currently maps `dist/` to `/data/www/app.jorlan.com/html/`
(for a separate nginx layout). This must be changed:

1. Remove the existing `linuxPackageMappings` entry that targets `/data/www/...`
2. Add a new mapping: `dist/` → `/opt/jorlan/www/`  
   (this is under `Linux / defaultLinuxInstallLocation = "/opt"` so it is owned by the `jorlan` user)
3. The default `application.conf` template must set `jorlan.web.root = "/opt/jorlan/www"`

No nginx configuration or separate web server is needed.

For local development:
- `sbtn web/debugDist` populates `debugDist/`
- Configure `jorlan.web.root = "debugDist"` in local `application.conf`

A helper script `scripts/build-web.sh` should be added for convenience:
```bash
#!/usr/bin/env bash
set -euo pipefail
cd "$(dirname "$0")/.."
cd stLib && sbt --error publishLocal && cd ..
sbt --error web/dist
```

## Auth Flow

Identical to dmscreen:
1. `AuthClient.whoami` on mount; if `None` → render `LoginRouter`
2. `LoginRouter` (from `zio-auth`) handles the login form + OAuth buttons
3. On success, the auth cookie is set by the server; page reload completes login
4. Logout: call `/api/auth/logout` then reload

## Key Risks

| Risk | Mitigation |
|------|------------|
| ScalablyTyped MUI v9 binding quality | v9 is quite new; run `stLib/publishLocal` and try a basic MUI Button first before writing pages |
| WebSocket subscriptions in ScalaJS | Follow dmscreen `ScalaJSClientAdapter` pattern exactly |
| Caliban client codegen | Must run `server/calibanGenClient` after any schema change |
| stLib out-of-date in CI | Add explicit `cd stLib && sbt publishLocal` step before main build |
